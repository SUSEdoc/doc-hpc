5 Slurm #

  • File Name: slurm.xml
  • ID: cha-slurm

Slurm is a workload manager for managing compute jobs on high performance computing cluster. It can start multiple jobs on a single node or a single job on multiple nodes. Additional components can be used for advanced scheduling and accounting.

The mandatory components of Slurm is the control daemon slurmctld taking care of job scheduling and the slurm daemon slurmd responsible for launching compute jobs. Subsequently nodes running the control daemon are called master nodes and nodes running the slurmd are called compute nodes.

Additional components are a secondary slurmctld acting as standby server for a failover and the slurm database daemon slurmdbd which stores the job history and user hirachy.

5.1 Slurm — Utility for HPC Workload Management #

  • File Name: slurm.xml
  • ID: sec-scheduler-slurm

For a minimal setup of Slurm with a controll node and multiple cmpute nodes, follow these instructions:

5.1.1 Minimal installation #

  • File Name: slurm.xml
  • ID: sec-slurm-minimal
Important: Make Sure of Consistent UIDs and GIDs for Slurm's Accounts

For security reasons, Slurm does not run as the user root but under its own user. It is important that the user slurm has the same UID/GID across all nodes of the cluster.

If this user/group does not exist, the package slurm creates this user and group when it is installed. However, this does not guarantee that the generated UIDs/GIDs will be identical on all systems.

Therefore, we strongly advise you to create the user/group slurm before installing slurm. If you are using a network directory service such as LDAP for user and group management, you can use it to provide the slurm user/group as well.

  1. It is strongly recommended that all compute nodes share a common home directory.

    Configure, enable and start munge on the control and compute nodes as described in Section 3.6, “munge Authentication”.

  2. Install slurm-node on the compute nodes with zypper in slurm-node and slurm on the controll node with zypper in slurm. The package mungewill be installed as dependence automatically.

Procedure 5.1: Creating slurm.conf #

  1. Edit the main configuration file /etc/slurm/slurm.conf:

    1. Configure the parameter ControlMachine=CONTROL_MACHINE with the host name of the control node.

      To find out the correct host name, run hostname -s on the control node.

    2. In order to define the compute nodes add: 

      NodeName=NODE_LIST  State=UNKNOWN

      where the actual parameters like CPUs for compute node can be obtained by running the following command on the compute node: 

      slurmd -C

      Additionally the line 

      PartitionName=normal Nodes=NODE_LIST \
  Default=YES MaxTime=24:00:00 State=UP

      has to be added, where NODE_LIST is the list of compute nodes (that is, the output of hostname -s run on each compute node (either comma-separated or as ranges: node[1-100]).

    3. Now copy the modified configuration file /etc/slurm/slurm.conf to the control node and all compute nodes: 

      scp /etc/slurm/slurm.conf COMPUTE_NODE:/etc/slurm/

  2. On the control node, start slurmctld: 

    systemctl start slurmctld.service

    Also enable it so that it starts on every boot: 

    systemctl enable slurmctld.service

  3. On the compute nodes, start and enable slurmd: 

    systemctl start slurmd.service
systemctl enable slurmd.service

    The last line causes slurmd to be started on every boot automatically.

Procedure 5.2: Testing the Installation #

  1. Check the status and availability of the compute nodes with the sinfo command. It should give you the following output in this example: 

    PARTITION AVAIL  TIMELIMIT  NODES  STATE NODELIST 
normal*      up 1-00:00:00      2   idle sle15sp2-node[1-2]

    If the node state is not idle see Section 5.3.1, “Frequently asked questions”.

  2. Now the slurm installation can be tested by running: 

    srun sleep 30

    This will try to immediatelly run the sleep on a free compute node. In another shell you can now [1] run the command 

    squeue

    It which will show you the running compute job in an output like 

        JOBID PARTITION     NAME     USER ST       TIME  NODES NODELIST(REASON) 
        1    normal    sleep     root  R       0:05      1 node02

  3. Now you can create the simple shell script 

    #!/bin/bash
echo "started at $(date)"
sleep 30
echo "finished at $(date)"

    save it as sleeper.sh and run the shell script in the queue with 

    sbatch sleeper.sh

    The shell script is executed as soon as enough resources are available. After the execution the output of the shell script is stored in the output file slurm-${JOBNR}.out.

5.1.2 Install slurm database #

  • File Name: slurm.xml
  • ID: sec-slurm-slurmdbd

With the minimal installation slurm will only store pending and running jobs. In order to store to finished and failed job data, the storage plugin has to be installed and enabled. Additionally so called completely fair scheduling can be enable, which replaces the FIFO [2] scheduling with algorithms which calculate the job priority in a queue in dependence of the job which a user has run in the history.

The slurm database has two components which is the daemon slurmdbd itself and a mysql database, where mariab is recommended. The database can be on the same node which runs slurmdbd or another node. For this simple setup all these services run on the same node.

Procedure 5.3: Install slurmdbd #
Note: Mariadb

If you want to use an external SQL database (or such a database is already installed on the control node), you can skip the Mariadb-related steps.

  1. Install the Mariabd SQL database with zypper in mariadb.

  2. Start and enable Mariadb with the following commands: 

    systemctl start mariadb
systemctl enable mariadb

  3. Now the database should be secure with the command mysql_secure_installation.

  4. In this step the database and the user for the slurm database has to be created. This is done by connecting to the SQL database with e.g. the command mysql -u root -p. After a successful connection the database and the creation of secure password [3] , the slurm user and the database is created with the commands 

    create user 'slurm'@'localhost' identified by 'password';
grant all on slurm_acct_db.* TO 'slurm'@'localhost';
create database slurm_acct_db;

  5. The package slurmdbd can be installed with the command 

    zypper in slurm-slurmdbd

    Now the configuration file /etc/slurm/slurmdbd.conf for slurmdbd has to be modified so that the daemon can access the database. Change the following lines 

    StoragePass=password

    to the password which you use in Step 4. When another location or user for the SQL database was chosen, you also need to modify the following entries: 

    StorageUser=slurm
DbdAddr=localhost
DbdHost=localhost

  6. The daemon slurmdbd should now be started and enabled with the commands 

    systemctl start slurmdbd
systemctl enable slurmdbd

    The first start of slurmdbd will take some time.

  7. To enable accounting, you have to have change/add the following lines for the connection between the slurmctld and the slurmdbd daemon: 

    JobAcctGatherType=jobacct_gather/linux
JobAcctGatherFrequency=30
AccountingStorageType=accounting_storage/slurmdbd
AccountingStorageHost=localhost

    In the example above, we assume that slurmdbd is running on the same host as slurmctld.

  8. Restart the slurmctld with: 

    systemctl restart slurmctld

  9. (Optional) Per default slurm does not take any group membership into account and it is not possible to map the system groups to slurm. The creation and group membership have to managed via the commandline tool sacctmgr. Its also possible to have a group hierarchy and users can be part of several groups.

    To create an umbrella group bavaria for two subgroups called nuremberg and munich use following commands 

    sacctmgr add account bavaria \
  Description="umbrella group for subgroups" Organization=bavaria
sacctmgr add account nuremberg,munich parent=bavaria 
Description="subgroup" Organization=bavaria

    With the similar syntax you can add now users to the accounts: 

    sacctmgr add user tux Account=nuremberg

5.2 Slurm administration commands #

  • File Name: slurm.xml
  • ID: sec-slurm-adm-commands

5.2.1 scontrol #

  • File Name: slurm.xml
  • ID: sec-slurm-sconfigure

The command scontrol is used to show and update the entities of slurm like the state of the compute nodes or compute jobs. It can also be used to reboot or propagate configuration changes to the compute nodes.

Useful options to this command are --details which will print a more verbose output and --oneliner forcing the output to a single what is useful for scripts.

scontrol show ENTITY

will display the state of the specified ENTITY.

scontrol update SPECIFICATION

can be used to update the SPECIFICATION like the compute node or compute node state.

Useful SPECIFICATION states for compute nodes which can be set are:

nodename=NODE state=drain reason=REASON

will drain the compute node so that no new jobs can be scheduled on the compute node, but will node end compute jobs running on the compute node. REASON could be any string.

nodename=NODE state=down reason=REASON

is removing all jobs from the compute node NODE, any jobs on the node will be aborted.

nodename=NODE state=drain reason=REASON

will drain the compute node so that no new jobs can be scheduled on the compute node, but will node end compute jobs running on the compute node. REASON could be any string.

The compute node will stay in drained state and must be put back to the idle state with the next listed command.

nodename=NODE state=resume

marks the compute node NODE to be ready for a return to the idle state.

jobid=JOBIDREQUIREMENT=VALUE

will update the given requirement like NumNodes with a new value. This command can also be executed as normal user.

reconfigure

will trigger a reload of the configuration slurm.conf on all compute nodes.

reboot NODELIST

can be used to reboot a compute node, as soon as the jobs on it have finished. The option RebootProgram="/sbin/reboot" have to be set in slurm.conf to use this command.

When the reboot of a compute node takes more than 60s, you set set an higher value for the parameter like ResumeTimeout=300 in slurm.conf.

sacctmgr

is used for job accounting within slurm. The service slurmdbd has to be setup in order to use this command. Follow the steps in Step 4 to setup slurmdbd.

sinfo

retrieves information about the state of the compute nodes and can be used for a fast overview of the cluster health. Following commandline switches are available

--dead

displays information about not responding nodes.

--long

use this switch for more detailed information.

--reservation

print information about advanced reservations.

-R

displays the reason why a node is in the down, drained or failing state.

--state=STATE

limit the output only to computes with the given state STATE.

sacct

displays, when accounting is enabled, the accounting data. Following options are available

--allusers

show accounting data for all users.

--accounts=NAME

only specified user(s) are shown

--starttime=MM/DD[/YY]-HH:MM[:SS]

only jobs after starttime will be showed. It also possible to have just MM/DD or HH:MM. If not time is given 00:00, what means that only job from this day are showed.

--endtime=MM/DD[/YY]-HH:MM[:SS]

same option as for --starttime. If not set, the time when the command was issued is used.

--name=NAME

limit output to jobs with the given NAME

--partition=PARTITION

show only job which run PARTITION.

sbatch, salloc and srun

this commands are used schedule a compute job, which is a batch script for sbatch, a interactive session for salloc and a binary for srun. Please note that only the command sbatch will place the submitted job script to the queue if it can't scheduled immediately.

Frequently used options for this commands are:

-n COUNT_THREADS

specifies the number of threads needed by the job. The threads can be allocated on different nodes.

-N MINCOUNT_NODES[-MAXCOUNT_NODES]

sets the number of compute nodes which are required for a job. The MAXCOUNT_NODES number can be omitted.

--time TIME

specifies the maximal clocktime (runtime) after which a job is killed. The format of TIME is either seconds or [HH:]MM:SS. No to be confused with the walltime which is the clocktime × threads.

--signal [B:]NUMBER[@TIME]

selects the signal with the NUMBER to be sent 60 seconds before job end if not TIME is specified. The signal will be sent to every process on every node. When only a signal should be sent to the controlling batch job, you have to specify the B: flag.

--job-name NAME

set the name of the job to NAME in the queue.

--array=RANGEINDEX

execute the given script via sbatch for indexes given by RANGEINDEX with the same parameters.

--dependency=STATE:JOBID

defer job until specified STATE of job JOBID has been reached.

--gres=GRES

run job only on node with the give GRES available. A generic resource is a GPU for example.

--licenses=NAME[:COUNT]

the job needs the number COUNT of licenses with the name NAME. In opposite to a gres, a license is not tight to a compute, but is a cluster wide variable.

--mem=MEMORY

sets the real memory MEMORY needed by a job per node, memory control has to be enabled in order to use this option. The default unit for MEMORY, but also K for kilobyte, M for megabyte, G for gigabyte and T for terabyte can be used.

--mem-per-cpu=MEMORY

same options as for --mem, but defines memory per used CPU.

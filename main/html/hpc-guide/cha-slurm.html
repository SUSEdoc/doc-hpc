5 Slurm #

  • File Name: slurm.xml
  • ID: cha-slurm

Slurm is a workload manager for managing compute jobs on high-performance computing clusters. It can start multiple jobs on a single node or a single job on multiple nodes. Additional components can be used for advanced scheduling and accounting.

The mandatory components of Slurm are the control daemon slurmctld, which takes care of job scheduling, and the slurm daemon slurmd, responsible for launching compute jobs. Subsequently, nodes running slurmctld are called master nodes and nodes running slurmd are called compute nodes.

Additional components are a secondary slurmctld acting as standby server for a failover, and the slurm database daemon slurmdbd, which stores the job history and user hierarchy.

5.1 Slurm — utility for HPC workload management #

  • File Name: slurm.xml
  • ID: sec-scheduler-slurm

For a minimal setup of Slurm with a control node and multiple compute nodes, follow these instructions:

5.1.1 Minimal installation #

  • File Name: slurm.xml
  • ID: sec-slurm-minimal
Important: Make sure of consistent UIDs and GIDs for Slurm's accounts

For security reasons, Slurm does not run as the user root but under its own user. It is important that the user slurm has the same UID/GID across all nodes of the cluster.

If this user/group does not exist, the package slurm creates this user and group when it is installed. However, this does not guarantee that the generated UIDs/GIDs will be identical on all systems.

Therefore, we strongly advise you to create the user/group slurm before installing slurm. If you are using a network directory service such as LDAP for user and group management, you can use it to provide the slurm user/group as well.

It is strongly recommended that all compute nodes share common user home directories. These should be provided through network storage.

  1. On the control node, install the slurm with the command zypper in slurm.

  2. On the compute nodes, install the slurm-node package with the command zypper in slurm-node.

  3. On both control and compute nodes, the package munge will be installed automatically.

    Configure, enable and start MUNGE on the control and compute nodes as described in Section 3.4, “MUNGE authentication”. Make sure that the same munge key is shared across all nodes.

Procedure 5.1: Creating slurm.conf #

  1. Edit the main configuration file /etc/slurm/slurm.conf:

    1. Configure the parameter ControlMachine=CONTROL_MACHINE with the host name of the control node.

      To find out the correct host name, run hostname -s on the control node.

    2. In order to define the compute nodes, add: 

      NodeName=NODE_LIST State=UNKNOWN

      This line allows specifying additional per-node parameters, such as Boards, SocketsPerBoard CoresPerSocket, ThreadsPerCore, or CPU. The actual values of these can be obtained by running the following command on the compute node: 

      slurmd -C

      Additionally, the line 

      PartitionName=normal Nodes=NODE_LIST \
  Default=YES MaxTime=24:00:00 State=UP

      has to be added, where NODE_LIST is the list of compute nodes (that is, the output of hostname -s run on each compute node (either comma-separated or as ranges: node[1-100]).

    3. Now copy the modified configuration file /etc/slurm/slurm.conf to the control node and all compute nodes: 

      scp /etc/slurm/slurm.conf COMPUTE_NODE:/etc/slurm/

  2. On the control node, start slurmctld: 

    systemctl start slurmctld.service

    Also enable it so that it starts on every boot: 

    systemctl enable slurmctld.service

  3. On the compute nodes, start and enable slurmd: 

    systemctl start slurmd.service
 systemctl enable slurmd.service

    The last line causes slurmd to be started on every boot automatically.

Procedure 5.2: Testing the installation #

  1. Check the status and availability of the compute nodes with the sinfo command. It should give you the following output in this example: 

    PARTITION AVAIL  TIMELIMIT  NODES  STATE NODELIST
normal*      up 1-00:00:00      2   idle sle15sp2-node[1-2]

    If the node state is not idle see Section 5.4.1, “Frequently asked questions”.

  2. Now the Slurm installation can be tested by running: 

    srun sleep 30

    This will try to immediately run the sleep on a free compute node. In another shell you can now [1] run the command: 

    squeue

    It will show you the running compute job in an output like this: 

        JOBID PARTITION     NAME     USER ST       TIME  NODES NODELIST(REASON)
        1    normal    sleep     root  R       0:05      1 node02

  3. Now you can create the simple shell script 

    #!/bin/bash
echo "started at $(date)"
sleep 30
echo "finished at $(date)"

    save it as sleeper.sh and run the shell script in the queue with 

    sbatch sleeper.sh

    The shell script is executed as soon as enough resources are available. After the execution the output of the shell script is stored in the output file slurm-${JOBNR}.out.

5.1.2 Install Slurm database #

  • File Name: slurm.xml
  • ID: sec-slurm-slurmdbd

With the minimal installation, Slurm will only store pending and running jobs. In order to store to finished and failed job data, the storage plugin has to be installed and enabled. Additionally, so-called completely fair scheduling can be enabled, which replaces FIFO [2] scheduling with algorithms which calculate the job priority in a queue in dependence of the job which a user has run in the history.

The Slurm database has two components: the slurmdbd daemon itself, and a SQL database, where MariaDB is recommended. The database can be on the same node which runs slurmdbd or another node. For this simple setup, all these services run on the same node.

Procedure 5.3: Install slurmdbd #
Note: MariaDB

If you want to use an external SQL database (or such a database is already installed on the control node), you can skip the MariaDB-related steps.

  1. Install the MariaDB SQL database with zypper in mariadb.

  2. Start and enable MariaDB with the following commands: 

    systemctl start mariadb
systemctl enable mariadb

  3. Now the database should be secure with the command mysql_secure_installation.

  4. In this step, the database and the user for the Slurm database have to be created. This is done by connecting to the SQL database, for example with the command mysql -u root -p. After a successful connection to the database and the creation of a secure password [3] , the Slurm user and the database are created with the commands: 

    create user 'slurm'@'localhost' identified by 'password';
grant all on slurm_acct_db.* TO 'slurm'@'localhost';
create database slurm_acct_db;

  5. The package slurmdbd can be installed with the command 

    zypper in slurm-slurmdbd

    Now the configuration file /etc/slurm/slurmdbd.conf for slurmdbd has to be modified so that the daemon can access the database. Change the following lines 

    StoragePass=password

    to the password which you used in Step 4. If you chose another location or user for the SQL database, you also need to modify the following entries: 

    StorageUser=slurm
DbdAddr=localhost
DbdHost=localhost

  6. The daemon slurmdbd should now be started and enabled with the commands 

    systemctl start slurmdbd
systemctl enable slurmdbd

    The first start of slurmdbd will take some time.

  7. To enable accounting, you have to have change/add the following lines for the connection between the slurmctld and the slurmdbd daemon: 

    JobAcctGatherType=jobacct_gather/linux
JobAcctGatherFrequency=30
AccountingStorageType=accounting_storage/slurmdbd
AccountingStorageHost=localhost

    In the example above, we assume that slurmdbd is running on the same host as slurmctld.

  8. Restart the slurmctld with: 

    systemctl restart slurmctld

  9. (Optional) By default, Slurm does not take any group membership into account, and it is not possible to map the system groups to Slurm. Group creation and membership have to be managed via the command line tool sacctmgr. It is also possible to have a group hierarchy, and users can be part of several groups.

    Note

    When using slurmdbd make sure a table for a cluster is added to the database before slurmctld is started (or restart it afterwards). Otherwise, no accounting information may be written to the database. To add a cluster table, run: sacctmgr -i add cluster CLUSTERNAME.

    To create an umbrella group bavaria for two subgroups called nuremberg and munich, use the following commands: 

    sacctmgr add account bavaria \
  Description="umbrella group for subgroups" Organization=bavaria
sacctmgr add account nuremberg,munich parent=bavaria
Description="subgroup" Organization=bavaria

    With the similar syntax you can add now users to the accounts: 

    sacctmgr add user tux Account=nuremberg

  • File Name: slurm.xml

5.2.1 scontrol #

  • File Name: slurm.xml
  • ID: sec-slurm-sconfigure

The command scontrol is used to show and update the entities of Slurm, such as the state of the compute nodes or compute jobs. It can also be used to reboot or to propagate configuration changes to the compute nodes.

Useful options to this command are --details, which will print more verbose output, and --oneliner, which forces the output onto a single line, which is more useful in shell scripts.

scontrol show ENTITY

will display the state of the specified ENTITY.

scontrol update SPECIFICATION

can be used to update the SPECIFICATION like the compute node or compute node state.

Useful SPECIFICATION states for compute nodes which can be set are:

nodename=NODE state=drain reason=REASON

will drain the compute node so that no new jobs can be scheduled on the compute node, but will node end compute jobs running on the compute node. REASON could be any string.

nodename=NODE state=down reason=REASON

is removing all jobs from the compute node NODE, any jobs on the node will be aborted.

nodename=NODE state=drain reason=REASON

will drain the compute node so that no new jobs can be scheduled on the compute node, but will node end compute jobs running on the compute node. REASON could be any string.

The compute node will stay in drained state and must be put back to the idle state with the next listed command.

nodename=NODE state=resume

marks the compute node NODE to be ready for a return to the idle state.

jobid=JOBIDREQUIREMENT=VALUE

will update the given requirement, such as NumNodes, with a new value. This command can also be executed as a normal user.

reconfigure

will trigger a reload of the configuration file slurm.conf on all compute nodes.

reboot NODELIST

can be used to reboot a compute node, as soon as the jobs on it have finished. The option RebootProgram="/sbin/reboot" have to be set in slurm.conf to use this command.

When the reboot of a compute node takes more than 60 seconds, you can set a higher value for the parameter, such as ResumeTimeout=300 in slurm.conf.

sacctmgr

is used for job accounting within Slurm. The service slurmdbd has to be setup in order to use this command. Follow the steps in Step 4 to setup slurmdbd.

sinfo

retrieves information about the state of the compute nodes, and can be used for a fast overview of the cluster health. The following command-line switches are available:

--dead

displays information about unresponsive nodes.

--long

shows more detailed information.

--reservation

prints information about advanced reservations.

-R

displays the reason why a node is in the down, drained, or failing state.

--state=STATE

limit the output only to nodes with the specified STATE state.

sacct

when accounting is enabled, displays the accounting data. The following options are available:

--allusers

show accounting data for all users.

--accounts=NAME

only the specified user(s) are shown

--starttime=MM/DD[/YY]-HH:MM[:SS]

only jobs after starttime will be shown. You can use just MM/DD or HH:MM. If no time is given, defaults to 00:00, which means that only jobs from today are shown.

--endtime=MM/DD[/YY]-HH:MM[:SS]

accepts the same options as for --starttime. If not set, the time when the command was issued is used.

--name=NAME

limit output to jobs with the given NAME

--partition=PARTITION

show only jobs which run in PARTITION.

sbatch, salloc and srun

these commands are used to schedule compute jobs, which means batch scripts for the sbatch command, interactive sessions for the salloc command, or binaries for the srun command.

Note that if the job cannot be scheduled immediately, only sbatch will place it into the queue.

Frequently-used options for these commands are:

-n COUNT_THREADS

specifies the number of threads needed by the job. The threads can be allocated on different nodes.

-N MINCOUNT_NODES[-MAXCOUNT_NODES]

sets the number of compute nodes which are required for a job. The MAXCOUNT_NODES number can be omitted.

--time TIME

specifies the maximal clocktime (runtime) after which a job is killed. The format of TIME is either seconds or [HH:]MM:SS. Not to be confused with walltime, which is clocktime × threads.

--signal [B:]NUMBER[@TIME]

means that the signal specified by NUMBER will be sent 60 seconds before the end of the job, unless TIME was specified. The signal will be sent to every process on every node. If a signal should only be sent to the controlling batch job, you must specify the B: flag.

--job-name NAME

set the name of the job to NAME in the queue.

--array=RANGEINDEX

execute the given script via sbatch for indexes given by RANGEINDEX with the same parameters.

--dependency=STATE:JOBID

defer job until specified STATE of job JOBID has been reached.

--gres=GRES

run a job only on nodes with the specified generic resource (GRes), for example a GPU, specified by the value of GRES.

--licenses=NAME[:COUNT]

the job must have the number specified in COUNT of licenses with the name NAME. A license is the opposite of a generic resource: it is not tied to a computer, but is a cluster-wide variable.

--mem=MEMORY

sets the real memory MEMORY needed by a job per node. To use this option, memory control must be enabled. The default unit for the MEMORY value is megabytes, but you can also use K for kilobyte, M for megabyte, G for gigabyte, and T for terabyte.

--mem-per-cpu=MEMORY

This takes the same options as --mem, but defines memory on a per-CPU basis rather than a per-node basis.

5.3 Upgrading Slurm #

  • File Name: slurm.xml
  • ID: sec-slurm-upgrade

New major versions of Slurm are released in regular intervals. With some restrictions (see below), interoperability is guaranteed between three consecutive versions. However, unlike updates to maintenance releases (that is, releases which differ in the last version number), upgrades to later major versions may require more careful planning.

For existing products under general support, version upgrades of Slurm are provided regularly. Unlike maintenance updates, these upgrades will not be installed automatically using zypper patch but require the administrator to request their installation explicitly. This is to ensure that these upgrades are not installed unintentionally and gives the administrator the opportunity to plan version upgrades beforehand.

On new installations, we recommend installing the latest available version.

Slurm uses a segmented version number: The first two segments denote the major version, the final segment denotes the patch level.

Check the list below for available major versions.

Upgrade packages (that is, packages that were not initially supplied with the module or service pack) have their major version encoded in the package name (with periods . replaced by underscores _). For example, for version 18.08, this would be: slurm_18_08-*.rpm.

To upgrade the package slurm to 18.08, run the command: 

zypper install --force-resolution slurm_18_08

To upgrade Slurm subpackages, use the analogous commands.

In addition to the “three major-version rule” mentioned at the beginning of this section, obey the following rules regarding the order of updates:

  1. The version of slurmdbd must be identical to or higher than the version of slurmctld

  2. The version of slurmctld must the identical to or higher than the version of slurmd

  3. The version of slurmd must be identical to or higher than the version of the slurm user applications.

Or in short:

version(slurmdbd) >= version(slurmctld) >= version(slurmd) >= version (Slurm user CLIs).

With each version, configuration options for slurmctld/slurmd or slurmdbd may be deprecated. While deprecated they will remain valid for this version and the two consecutive ones but they may be removed later.

Therefore, it is advisable to update the configuration files after the upgrade and replace deprecated configuration options before the final restart of a service.

It should be noted that a new major version of Slurm introduces a new version of libslurm. Older versions of this library might no longer work with an upgraded Slurm. For all SLE software depending on libslurm , an upgrade will be provided. Any locally-developed Slurm modules or tools may require modification and/or recompilation.

5.3.1 Upgrade workflow #

  • File Name: slurm.xml
  • ID: sec-slurm-upgrade-workflow

For this workflow, we assume that MUNGE authentication is used and that pdsh, the pdsh Slurm plugin and mrsh are usable to access the all machines of the cluster (that is mrshd is running on all nodes in the cluster).

If this is not the case, install pdsh: 

# zypper in pdsh-slurm

if mrsh is not used in the cluster, the ssh back-end for pdsh may be used as well for this, simply replace the option -R mrsh by -R sshin the pdshcommands below. This is less scalable and you may run out of usable ports.

Procedure 5.4: Upgrading Slurm #

  1. Upgrade slurmdbd Database Daemon

    If the database daemon slurmdbd is used, it must be upgraded first. Care must be taken if the same database is used for multiple clusters. The database needs to be updated before any cluster is updated.

    It should be noted that when upgrading slurmdbd, a conversion of the database will take place when the new version of slurmdbd is started for the first time. If the database is big the conversion will take several tens of minutes. During this time, the database will be inaccessible.

    It is highly recommended to create a backup of the database in case an error occurs during the upgrade process or afterwards: without a backup all accounting data collected in the database may be lost if an error occurs or the upgrade must be rolled back for some reason. A database converted to a newer version cannot be converted back to an older one and older versions of slurmdbd will not recognize the newer formats. To backup and upgrade slurmdbd you may follow this procedure:

    1. Stop the slurmdbd service: 

      # rcslurmdbd stop

      Make sure that slurmdbd is not running anymore: 

      # rcslurmdbd status

      It should be noted that slurmctld might remain running while the database daemon is down. While it is down, requests intended for slurmdbd are queued internally. The DBD Agent Queue size is limited, however, and should therefore be monitored with sdiag.

    2. Create a backup of the slurm_acct_db database: 

      # mysqldump -p slurm_acct_db > slurm_acct_db.sql

      If needed, this can be restored calling: 

      # mysql -p slurm_acct_db < slurm_acct_db.sql

    3. In preparation of the conversion, make sure, the variable innodb_buffer_size is set to a value >= 128 Mb:

      On the database server, run: 

      # echo  'SELECT @@innodb_buffer_pool_size/1024/1024;' | \
  mysql --password --batch

      If the size is less than 128 MB, it can be changed on the fly for the current session (on mariadb) 

      # echo 'set GLOBAL innodb_buffer_pool_size = 134217728;' | \
  mysql --password --batch

      or permanently by editing /etc/my.cnf, setting it to 128 MB then restarting the database: 

      # rcmysql restart

    4. Install the upgrade of slurmdbd: 

      zypper install --force-resolution slurm_version-slurmdbd
      Note: Update MariaDB separately

      If you also need to update mariadb, it is recommended to perform this step separately, before performing step Step 1.d.

      1. Upgrade MariaDB: 

        # zypper update mariadb

      2. Run the conversion of the database tables to the new version of MariaDB: 

        mysql_upgrade --user=root --password=root_db_password;

    5. Rebuild database

      Since a conversion may take a considerable amount of time, the systemd service may run into a timeout during the conversion. Thus we recommend performing the migration manually, by running slurmdbd from the command line in the foreground: 

      # /usr/sbin/slurmdbd -D -v

      Once you see the message: 

      Conversion done:
success!

      slurmdbd may be shut down with signal SIGTERM (that is, by pressing Ctrl–C).

      When using a backup slurmdbd, the conversion needs to be performed on the primary. The backup will start after the conversion has completed.

    6. Before restarting the service, you should remove or replace any deprecated configuration options. Check the list of deprecated options below. After this has been completed, restart slurmdbd: 

      # systemctl start slurmdbd
      Note: No daemonization during rebuild

      During the rebuild of the Slurm database, the database daemon does not daemonize.

      Note: Convert primary slurmdbd first

      If a backup slurmdbd daemon is used, the primary slurmdbd needs to be upgraded and run to perform the conversion first. The backup slurmdbd will not start until this has happened.

  2. Update slurmctld and slurmd

    Once the Slurm database has been updated, the slurmctld and slurmd instances may be updated. It is recommended to update the head and compute nodes all in a single pass. If this is not feasible, the compute nodes (slurmd) may be updated on a node-by-node basis. However, this requires that the master nodes (slurmctld) have been updated successfully.

    1. Back up the slurmctld/slurmd configuration

      It is advisable to create a backup copy of the Slurm configuration before starting the upgrade process. Since the configuration file /etc/slurm/slurm.conf should be identical across the entire cluster, it is sufficient to do so on the master controller node.

    2. Increase Timeouts

      Set SlurmdTimeout and SlurmctldTimeout in /etc/slurm/slurm.conf to sufficiently high values to avoid timeouts while slurmctld and slurmd are down. We recommend at least 60 minutes, and more on larger clusters.

      1. Edit /etc/slurm/slurm.conf on the master controller node and set the values for this variable to at least 3600 (one hour). 

        SlurmctldTimeout=3600
SlurmdTimeout=3600

      2. Copy /etc/slurm/slurm.conf to all nodes, if MUNGE authentication is used in the cluster as recommended.

        1. Obtain the list of partitions in /etc/slurm/slurm.conf.

        2. Execute: 

          # cp /etc/slurm/slurm.conf /etc/slurm/slurm.conf.update
# sudo -u slurm /bin/bash -c 'cat /etc/slurm/slurm.conf.update \
  | pdsh -R mrsh -P partitions \
  "cat > /etc/slurm/slurm.conf"'
# rm /etc/slurm/slurm.conf.update
# scontrol reconfigure

        3. Verify that the reconfiguration took effect: 

          # scontrol show config | grep Timeout

    3. Shut down any running slurmctld instances

      1. If applicable, shut down any backup controllers on the backup head nodes: 

        backup: # systemctl stop slurmctld

      2. Shut down the master controller: 

        master: # systemctl stop slurmctld

    4. Back up the slurmctld state files

      Also, it should be noted, that slurmctld maintains state information that is persistent. Almost every major version involves changes to the slurmctld state files. This state information will be upgraded as well if the upgrade remains within the supported version range and no data will be lost.

      However, if a downgrade should become necessary, state information from newer versions will not be recognized by an older version of slurmctld and thus will be discarded, resulting in a loss of all running and pending jobs. Thus it is useful to back up the old state in case an update needs to be rolled back.

      1. Determine the StateSaveLocation directory: 

        # scontrol show config | grep StateSaveLocation

      2. Create a backup of the content of this directory to be able to roll back the update if an issue arises.

        Should a downgrade be required make sure to restore the content of the StateSaveLocation directory from this backup.

    5. Shut down slurmd on the nodes 

      # pdsh -R ssh -P partitions systemctl stop slurmd

    6. Update slurmctld on the master and backup nodes as well as slurmd on the compute nodes

      1. On the master/backup node(s): run: 

        master: # zypper zypper install \
  --force-resolution slurm_version

      2. On the master node run: 

        master: # pdsh -R ssh -P partitions \
  zypper install --force-resolution \
  slurm_version-node

    7. Replace deprecated options

      If deprecated options are to be replaced in the configuration files (see list below), this may be performed before updating the services. These configuration files may be distributed to all controllers and nodes of the cluster by using the method as described in 3.b.bb above.

      Note: Memory size seen by slurmd may change on update

      Under certain circumstances, the amount of memory seen by slurmd may change after an update. If this happens, slurmctld will put the nodes in a drained state. To check, whether the amount of memory seem by slurmd will change after the update, you may run on a single compute node: 

      node1: # slurmd -C

      Compare the output with the settings in slurm.conf. If required, correct the setting.

    8. Restart slurmd on all compute nodes

      On the master controller run: 

      master: # pdsh -R ssh -P partitions \
  systemctl start slurmd

      On the master, run: 

      master: # systemctl start slurmctld

      then execute the same on the backup controller(s)

    9. Verify if the system operates properly

      1. Check the status of the controller(s). On the master and backup controllers, run: 

        # systemctl status slurmctld

      2. and verify that the services are running without errors.

        Run: 

        sinfo -R

        to check whether there are any down, drained, failing or failed nodes after restart.

    10. Clean up

      Restore the SlurmdTimeout and SlurmctldTimeout values in /etc/slurm/slurm.conf on all nodes and run scontrol reconfigure (see Step 2.b).

A new major version of libslurm is provided with each service pack of SUSE Linux Enterprise High Performance Computing. The old version will not be uninstalled on upgrade, and user-provided applications built with an old version should continue to work if the old library used is not older than the past two versions. It is strongly recommended to rebuild local applications using libslurm — such as MPI libraries with Slurm support — as early as possible. This may require updating the user applications, as new arguments may be introduced to existing functions.

5.4.1 Frequently asked questions #

  • File Name: slurm.xml
  • ID: sec-slurm-faq
Q: 1. How do I change the state of a node from down to up?

When the slurmd daemon on a node does not reboot in the time specified in the ResumeTimeout parameter, or the ReturnToService was not changed in the configuration file slurm.conf, compute nodes stay in the down state and have to be set back to the up state manually. This can be done for the NODE with the following command: 

scontrol update state=resume NodeName=NODE
Q: 2. What is the difference between the states down and down*?

A * shown after a status code means that the node is not responding.

Thus, when a node is marked as down*, it means that the node is not reachable due to network issues, or that slurmd is not running on that node.

In the down state, the node is reachable, but either the node was rebooted unexpectedly, the hardware does not match the description in slurm.conf, or a health check was configured with the HealthCheckProgram.

Q: 3. How do I get the exact core count, socket number and number of CPUs for a node?

The values for a node which go into the configuration file slurm.conf can be obtained with the command: 

slurmd -C
